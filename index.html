用户和用户组管理

发表于 2020-04-30

用户

新建用户

useradd {用户名}

没有指定组时，会自动创建一个同用户名的组

查看用户

id {用户名}

​ 不提供用户名时查看当前用户

root 用户的家目录是 /root

其他用户的家目录是 /home/用户名

修改密码

passwd [用户名]

没有指定用户时修改当前用户的密码

删除用户

userdel {用户名} [-r]

加 -r 表示将配置文件也删除，包括 /home/用户名 文件夹

配置用户

  • usermod
    • -d {目录} {用户名} 设定家目录
    • -g {组名} {用户名} 修改用户到用户组
    • …

chage

…

用户组

新建用户组

groupadd {组名}

修改用户到用户组

usermod -g {组名} {用户名}

在创建用户时添加到组

useradd -g {组名} {用户名}

切换用户

  • su - {用户名} 切换用户

  • su {用户名} 不完全切换，目录没有移动到用户的家目录

    普通用户之间切换需要密码, root 切换到普通用户不需要密码

vim command

常用命令

  • w 保存文件，vim打开时没有指定文件需要提供文件名: w filename
  • q 退出
  • q! 强制退出
  • !command 执行 shell 命令 ( command )
  • /word 查找 word
    • n 查找下一个
    • shift + n 查找上一个
  • s/before/after 将光标所在行的 before 替换为 after
  • %s/before/after 将文件中的 before 替换为 after
  • %s/before/after/g 将文件中的所有 before 替换为 after ( global )
  • x, ys/before/after[/g] 将第 x 行到第 y 行的 [所有] before 替换为 after
  • set nu / nonu 打开 / 关闭行号
  • set hlsearch / nohlsearch 打开 / 关闭搜索高亮

NORMAL 模式下：

  • i 在光标处进入插入模式
  • I 光标移至当前行首位，进入插入模式
  • a 光标后移一位，进入插入模式

复制，剪切和粘贴

复制

NORMAL 模式下：

  • y 在使用 v 模式选定区域时，复制选定块
  • yy 复制整行
  • nyy / yny 从当前光标向下复制 n 行
  • y^ 复制当前行光标到首位的内容
  • y$ 复制当前行光标到末位的内容

  • yG 复制当前光标到文件末位的内容
  • nyG / ynG 将光标移至第 n 行，再复制到文件末位的内容

剪切

  • dd 剪切整行

    整行粘贴时在光标 上 / 下，非整行粘贴时在光标 前 / 后

命令前面加数字表示重复的次数，加字母表示使用的缓冲区名称。使用英文句号 . 可以重复上一个命令。

NORMAL 模式下：

  • ctrl + r 重做

  • 参与制订 URI 规范（1998.8）
  • 主导制订 HTTP/1.1 规范（1999.6）
  • 2000 年发布指导 HTTP/1.1 规范制订的论文
    • 《Architectural Styles and the Design of Network-based Software Architectures》（架构风格与基于网络的软件架构设计），即 Representational State Transfer（REST）架构。
  • Apache 基金会（The Apache Software Foundation）共同创始人。
    • 参与开发 Apache httpd 服务。

Form Follows Function

  • HTTP 协议
    • Roy Thomas Fielding：HTTP 主要作者，REST 架构作者。
  • URI：统一资源标识符

HTTP 解决了什么问题

解决 WWW 信息交互必须面对的需求：

  • 低门槛

    • 例子：java applet 门槛过高

  • 可扩展性：巨大的用户群体，超长的寿命

  • 分布式系统下的 Hypermedia：大粒度数据的网络传输

  • Internet 规模

    • 无法控制的 scalability
      • 不可预测的负载、非法格式的数据、恶意消息
      • 客户端不能保证所有服务器信息，服务器不能保持多个请求间的状态信息。
    • 独立的组件部署：新老组件并存

  • 向前兼容：自 1993 年起 HTTP0.9\1.0（1996）已经被广泛使用。

OSI模型与TCP-IP模型

发表于 2020-04-29

OSI ( Open System Interconnection Reference Model ) 概念模型

开放式系统互联模型（英语：Open System Interconnection Model，缩写：OSI；简称为OSI模型）是一种概念模型，由国际标准化组织提出，一个试图使各种计算机在世界范围内互连为网络的标准框架。定义于ISO/IEC 7498-1。

第7层 应用层

应用层（Application Layer）提供为应用软件而设的接口，以设置与另一应用软件之间的通信。例如: HTTP、HTTPS、FTP、TELNET、SSH、SMTP、POP3、HTML等。

第6层 表达层

表达层（Presentation Layer）把数据转换为能与接收者的系统格式兼容并适合传输的格式。

第5层 会话层

会话层（Session Layer）负责在数据传输中设置和维护计算机网络中两台计算机之间的通信连接。

第4层 传输层

传输层（Transport Layer）把传输表头（TH）加至数据以形成数据包。传输表头包含了所使用的协议等发送信息。例如:传输控制协议（TCP）等。

第3层 网络层

网络层（Network Layer）决定数据的路径选择和转寄，将网络表头（NH）加至数据包，以形成报文。网络表头包含了网络数据。例如:互联网协议（IP）等。

第2层 数据链接层（数据链路层）

数据链路层（Data Link Layer）负责网络寻址、错误侦测和改错。当表头和表尾被加至数据包时，会形成信息框（Data Frame）。数据链表头（DLH）是包含了物理地址和错误侦测及改错的方法。数据链表尾（DLT）是一串指示数据包末端的字符串。例如以太网、无线局域网（Wi-Fi）和通用分组无线服务（GPRS）等。

分为两个子层：逻辑链路控制（logical link control，LLC）子层和介质访问控制（Media access control，MAC）子层。

第1层 物理层

物理层（Physical Layer）在局部局域网上传送数据帧（Data Frame），它负责管理电脑通信设备和网络媒体之间的互通。包括了针脚、电压、线缆规范、集线器、中继器、网卡、主机接口卡等。

TCP-IP 模型

第4层 应用层

例如HTTP、FTP、DNS

（如 BGP 和 RIP 这样的路由协议，尽管由于各种各样的原因它们分别运行在 TCP 和 UDP 上，仍然可以将它们看作网络层的一部分）

第3层 传输层

例如TCP、UDP、RTP、SCTP
（如OSPF这样的路由协议，尽管运行在IP上也可以看作是网络层的一部分）

第2层 网络互连层

对于TCP/IP来说这是因特网协议（IP）
（如 ICMP和 IGMP 这样的必须协议尽管运行在 IP 上，也仍然可以看作是网络互连层的一部分；ARP不运行在IP上）

第1层 网络访问(链接)层

例如以太网、Wi-Fi、MPLS等。

分层：

  • 好处：良好的封装，可扩展性，互不影响。

  • 网络互连层 ipv4 -> ipv6 进展缓慢，应用层飞速发展。

  • 坏处：数据延迟。

    • 为了解决性能问题，有 DPDK 这样的组件出现。绕过一些层以提高网络性能，某些系统的 debug 工具在开启后无法使用。

      DPDK（英语：Data Plane Development Kit），是一组快速处理数据包的开发平台及接口。运行于Intel X86与arm平台上（最新版本也开始支持PowerPC）。该平台采用BSD许可证发布。

基于ABNF语义定义的HTTP消息格式

发表于 2020-04-29

ABNF（扩充巴科斯-诺尔范式）

在计算机科学中，扩充巴科斯-瑙尔范式（ABNF）是一种基于巴科斯-瑙尔范式（BNF）的元语言，但它有自己的语法和派生规则。ABNF的原动原则是描述一种作为双向通信协议的语言的形式系统。它是由第68号互联网标准（”STD 68”，大小写样式按照原文）定义的，也就是RFC 5234，经常用于互联网工程任务组（IETF）通信协议的定义语言。

RFC 5234取代了RFC 4234 (取代了RFC 2234).

RFC 5234 - Augmented BNF for Syntax Specifications: ABNF

操作符

  • 空白字符: 用来分隔定义中的各个元素

    • method SP request-target SP HTTP-version CRLF

  • 选择 /：表示多个规则都是可供选择的规则

    • start-line = request-line / status-line

  • 值范围 %c##-##：

    • OCTAL = “0” / “1” / “2” / “3” / “4” / “5” / “6” / “7” 与 OCTAL = %x30-37 等价

  • 序列组合 ()：将规则组合起来，视为单个元素

  • *不定量重复 *m*n：

    • * 元素表示零个或更多元素：*(header-field CRLF)
    • 1* 元素表示一个或更多元素，2*4 元素表示两个至四个元素

  • 可选序列 []：

    • [ message-body ]

核心规则

规则 形式定义 意义
ALPHA %x41-5A / %x64-7A 大写和小写ASCII字母（A-Z，a-z）
DIGIT %x30-39 数字（0-9）
HEXDIG DIGIT / “A” / “B” / “C” / “D” / “E” / “F” 十六进制数字（0-9，A-F，a-f）
DQUOTE %x22 双引号
SP %x20 空格
HTAB %x09 横向制表符
WSP SP / HTAB 空格或横向制表符
LWSP *(WSP / CRLF WSP) 直线空白（晚于换行）
VCHAR %x21-7E 可见（打印）字符
CHAR %x01-7F 任何7-位US-ASCII字符，不包括NUL（%x00）
OCTET %x00-FF 8位数据
CTL %x00-1F / %x7F 控制字符
CR %x0D 回车（MAC OS）
LF %x0A 换行 （Linux）
CRLF CR LF 互联网标准换行 （Windows）
BIT “0” / “1” 二进制数字

基于 ABNF 描述的 HTTP 协议格式

HTTP-message = start-line *( header-field CRLF) CRLF [ message-body ]

  • start-line = request-line / status-line

    • request-line = method SP request-target SP HTTP-version CRLF
    • status-line = HTTP-version SP status-code SP reason-phrase CRLF

  • header-field = field-name “:” OWS field-value OWS

    • OWS = *( SP / HTAB )
    • field-name = token
    • field-value = *( field-content / obs-fold )

  • message-body = *OCTET

收费的原因

收费规则

  • 50元（人民币）可租用机器人一年（366天）

机器人FAQ

发表于 2020-01-26

关于QQ机器人五十弦(strings-bot)的FAQ

Q：机器人是怎么做的？

A：基于酷Q和cqhttp 和 nonebot 的QQ机器人，具体实现代码已开源于GitHub。

Q：机器人是用什么语言写的？

A：python。

Q：机器人有什么用？

A：除了活跃之外，没用。

Q：机器人回复 “请求次数超限制!” 或只会回复 “？”是什么意思？

A：由于机器人使用了图灵机器人的接口（用于在接收到无法处理，即机器人不支持的指令时调用）。免费的图灵接口是有调用次数限制的，100条/天(现在有一千条了)，这就意味着在一天中机器人最多只能回复一百次一千次，这一百次一千次是对于机器人而言的（所以你可能今天一句话都没有和它说过但调用次数已经超限了）。当调用次数超限后机器人就无法调用图灵接口了，这时就会向你返回“请求次数超限制!”的提示。事实上，图灵机器人的对话只是可有可无的，在我原来的计划中并不存在。

Q：在使用过程中有疑问怎么办？

A：大多数的问题都可以在 strings-bot-README 中得到解决。如果还是有疑问可以向我的邮箱（jin_ser@qq.com）发送邮件。

Q：遇到bug怎么办？

A：可以在GitHub的机器人仓库中提交issue；如果你愿意，也欢迎提交pull requests。或者也可以向我的邮箱（jin_ser@qq.com）发送邮件。
